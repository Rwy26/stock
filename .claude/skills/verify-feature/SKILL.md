# 피처 검증 워크플로우 (verify-feature)

신규 피처를 DB→백엔드→프론트엔드 레이어에 걸쳐 실데이터로 검증한다.

## 설계 확인 (구현 전)

구현 시작 전 반드시 아래 질문을 확인한다:
1. DB 스키마 변경이 있는가? (마이그레이션 필요 여부)
2. 백엔드 엔드포인트 변경/추가 범위
3. 프론트엔드 연동 방식 (REST, 폴링, 웹소켓)
4. 실데이터 검증 기준 (기대 출력값, 예외 케이스)

## 구현 후 검증 순서

```
1. DB 레이어: 스키마 적용 확인, 샘플 쿼리로 데이터 형태 확인
2. 백엔드: 엔드포인트 smoke test (curl 또는 httpx)
3. 실데이터 검증:
   - NameError / null이 다운스트림을 무효화하지 않는지 assert
   - 실시장 데이터로 출력값 shape 확인
4. 프론트엔드: 렌더링 확인
```

## 완료 선언 조건
- 위 4단계 전부 통과 후에만 "완료" 보고
- 데이터 한계(샘플 수 부족 등)는 메트릭과 함께 명시

## 보고 형식
```
피처 검증 완료: [피처명]
  DB: ✓ (테이블: ...)
  백엔드: ✓ (엔드포인트: ...)
  실데이터: ✓ (샘플 N건, 출력 shape: ...)
  프론트엔드: ✓
  데이터 한계: (있으면 명시)
```
